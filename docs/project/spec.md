# Codex + MCP Local Dev Spec

**Project goal:** Build a real small website for the **Greater Gaming Society of San Antonio, TX** in a VS Code workspace and use **Codex** with **Playwright MCP** as the primary MCP to help implement and validate it locally.

---

## What this is

This is a starter spec for building a practical website in a normal local workflow.

The goal is not a throwaway demo. The goal is a small site that can grow into a real public-facing home for the organization.

Core workflow:

- **Codex** reads and edits files in the workspace
- **Playwright MCP** opens and validates the local site in a browser
- **Optional MCPs** may be added later only when they support a real workflow
- **VS Code** is the main editor/workspace

---

## Main outcome

By the end of setup, I want to be able to:

1. Open the project in VS Code

3. Use Codex in that workspace

## Event content model for v1

Treat events as structured content from the start, even though v1 is static-first.

Each event should be representable with fields such as:

- `title`
- `date`
- `time`
- `location`
- `description`
- `externalUrl`
- `status`

Recommended status values:

- `upcoming`
- `past`
- `cancelled`

For v1, store events in a simple content source inside the repo, such as a TypeScript data file. The public site should read from that content source instead of hardcoding event cards directly into page JSX.

This keeps the contract stable for a future admin interface.

## Playwright MCP usage

Playwright MCP should be treated as the primary browser validation tool.

Use it for things like:

- opening the local website
- checking layout in the browser
- validating hero text and CTA rendering
- checking organization/about content
- validating event cards or event listings
- checking contact links
- spotting broken interactions
- confirming responsive behavior

Codex edits files in the repo. Playwright MCP validates the running result.

## Next DevTools MCP usage

Next DevTools MCP should be treated as the Next.js-specific support layer.

Use it for things like:

- initializing Next.js context at the start of a session
- searching official Next.js docs
- understanding route structure
- checking runtime errors and dev logs on a running Next.js app
- inspecting what the Next.js dev server is exposing

When using this MCP, start the local app first and call the `init` tool early in the session.

## Context7 MCP usage

Context7 should be treated as the current documentation MCP for framework and library questions.

Use it for things like:

- looking up current Next.js APIs
- checking Tailwind usage patterns
- confirming Biome commands and configuration
- pulling current examples for React and related packages

Context7 is especially useful when you want version-aware documentation while building.

## Future admin planning

The event admin interface is a later phase, not part of v1.

Planned future shape:

- a protected route or separate small dashboard
- one page to list existing events
- one page or view for create/edit event
- ability to archive or cancel an event
- optional publish/unpublish state

Keep the admin intentionally narrow:

- no broad CMS ambition for v1
- no complex role system unless requirements expand
- no provider-specific backend assumption yet

Using Next.js now keeps the path cleaner if the project later adds authenticated routes, server-side writes, or event-management workflows.

The important requirement today is to keep the public event-rendering code decoupled from the storage layer so the site can move from repo-managed content to admin-managed content later.
